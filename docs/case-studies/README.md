## What a good case study looks like

Not "we achieved great results using `autogap`". That's marketing, not a case study.

A good case study:

1. Names the **objective** the project was trying to reach, and why the time was running out.
2. Shows the **three gaps** the skill surfaced and *why those three out of the N that existed*.
3. Shows the **three macro-steps** and how they were ordered.
4. Tells you what **actually happened**, including the surprises and the recovery.
5. Gives you **artifacts** when possible: commit SHAs, screenshots, Lighthouse JSON, before/after KPIs.
6. Names **what the skill got wrong or could have done better** — if the case doesn't have any such note, it's either flawless (unlikely) or whitewashed (likely).

## Contributing a case study

See the main [`CONTRIBUTING.md`](../../CONTRIBUTING.md). Case studies from real projects are the single highest-value contribution this repo accepts.

Acceptable:

- Fully public, with the owner's consent and real names.
- Anonymized, with sector and order-of-magnitude preserved; mark as `DRAFT` until you've verified the metrics are safe to share.
- A failure case is as valuable as a success case. `autogap` misclassified a project, missed a gap, proposed something that didn't work — write that up.

Reach out via an issue before you write a long one, so we can help shape the framing.
